# PURPOSE

This document defines the canonical governance rules for the
Agile AI University shared shell architecture.

The shared shell provides:

- Shared header engine
- Shared footer engine
- Shared theme engine
- Shared institutional styling
- Shared navigation framework
- Shared refinement architecture

The purpose of this governance model is to ensure:

- Institutional consistency
- Cross-surface stability
- Structural predictability
- Long-term maintainability
- Surface isolation
- Controlled refinement evolution

This document is authoritative across all institutional surfaces.


# ==========================================================
# CANONICAL ARCHITECTURE MODEL
# ==========================================================

All institutional public surfaces operate using:

- Shared Design Authority Layer
- Config-Driven Navigation
- Surface-Specific Content
- Shared Structural Shell

The shell architecture is centralized and reused.

No public surface may independently fork or duplicate
shared shell infrastructure.


# ==========================================================
# AUTHORITATIVE SHARED COMPONENTS
# ==========================================================

Canonical shared shell assets:

/shared/design-authority/js/header.js
/shared/design-authority/js/footer.js
/shared/design-authority/js/theme.js
/shared/design-authority/css/site.css
/shared/design-authority/css/refinement.css

These files are governance-controlled assets.


# ==========================================================
# CONFIG-DRIVEN NAVIGATION MODEL
# ==========================================================

Navigation must be controlled via:

window.SURFACE_CONFIG

The shared shell must NEVER contain:

- Hardcoded surface branching
- Inline surface-specific navigation
- Conditional institutional logic
- Surface ownership assumptions

Each surface defines its own:

- Brand label
- Brand href
- Navigation structure
- Active path mappings

The shell remains structurally neutral.


# ==========================================================
# SURFACE ISOLATION RULE
# ==========================================================

Each public surface is fully self-contained.

Per architecture lock:

Each hosting target must maintain its own:

/shared/design-authority/

directory within its own hosting root.

Root-level shared assets are NOT runtime-authoritative.

Cross-surface runtime dependencies are prohibited.


# ==========================================================
# MODIFICATION GOVERNANCE
# ==========================================================

Shared shell modifications are HIGH-RISK changes.

Allowed modifications:

- Structural stabilization
- Accessibility improvements
- Mobile responsiveness fixes
- Governance-safe refinements
- Security hardening
- Cross-browser compatibility improvements

Prohibited modifications:

- Surface-specific logic
- Experimental UI divergence
- Inline style injection
- Ad-hoc component duplication
- Visual redesign without governance review
- Breaking structural compatibility


# ==========================================================
# REFINEMENT LAYER GOVERNANCE
# ==========================================================

Visual refinement must occur ONLY via:

refinement.css

The canonical site.css file represents:

- Core institutional structure
- Foundational spacing system
- Typography authority
- Shared shell behavior

site.css must NOT be modified for cosmetic experimentation.

All refinement changes must be:

- Additive
- Reversible
- Non-destructive
- Structurally neutral


# ==========================================================
# HEADER ENGINE GOVERNANCE
# ==========================================================

header.js responsibilities:

- Shared header injection
- Navigation rendering
- Mobile navigation control
- Active state management
- Accessibility attributes
- Shared structural behavior

header.js must NOT contain:

- Analytics logic
- Surface-specific branching
- Business logic
- Data loading
- Authentication logic
- Institutional content ownership


# ==========================================================
# FOOTER ENGINE GOVERNANCE
# ==========================================================

footer.js responsibilities:

- Shared footer injection
- Institutional governance links
- Shared compliance structure
- Institutional reference surfaces

footer.js must remain:

- Structurally lightweight
- Institutionally neutral
- Reusable across surfaces


# ==========================================================
# THEME ENGINE GOVERNANCE
# ==========================================================

theme.js responsibilities:

- Theme persistence
- Theme toggling
- Shared theme synchronization
- Dark/light mode handling

Theme logic must remain:

- Surface-neutral
- Lightweight
- Non-blocking
- Independent of content systems


# ==========================================================
# VERSIONING POLICY
# ==========================================================

All shared shell assets must maintain:

- Explicit version identifiers
- Governance comments
- Structured modification summaries
- Locked status indicators where applicable

Example:

Version: v4.1
Status: GOVERNANCE LOCKED

Modification comments must describe:

- What changed
- Why it changed
- Structural impact
- Governance implications


# ==========================================================
# DEPLOYMENT GOVERNANCE
# ==========================================================

Shared shell deployments must follow:

1. Local validation
2. Cross-surface validation
3. Mobile validation
4. Theme validation
5. Navigation validation
6. Deployment
7. Cache verification
8. Runtime validation

Deployment shortcuts are prohibited.


# ==========================================================
# VALIDATION CHECKLIST
# ==========================================================

Before deployment validate:

- Header renders correctly
- Footer renders correctly
- Mobile navigation works
- Theme toggle works
- Active states work
- No layout overlap
- No console errors
- No runtime injection failures
- Footer spacing stable
- Dark mode stable
- Cross-surface consistency maintained


# ==========================================================
# DESIGN PHILOSOPHY
# ==========================================================

The Agile AI University shell architecture prioritizes:

- Institutional clarity
- Structural stability
- Operational simplicity
- Governance discipline
- Long-term maintainability
- Surface differentiation
- Minimalism over visual noise

The shell exists to support institutional ecosystems,
not compete with content.


# ==========================================================
# SURFACE DIFFERENTIATION PRINCIPLE
# ==========================================================

Shared shell consistency does NOT mean
surface identity duplication.

Each surface may maintain distinct:

- Content architecture
- Operational intent
- Information density
- Experience positioning

Examples:

- EDU → Knowledge Surface
- LAB → Operational Environment
- PORTAL → Controlled Participant System
- VERIFY → Institutional Trust Surface

Differentiation should occur through content
and refinement layers — not shell fragmentation.
