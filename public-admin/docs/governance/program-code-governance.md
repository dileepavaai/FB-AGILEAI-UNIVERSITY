# Purpose

This document defines the governance framework for Program Codes within the Agile AI University credential ecosystem.

Program Codes serve as the canonical institutional identifiers for credential programs.

The purpose of Program Code governance is to ensure:

* Consistency
* Stability
* Traceability
* Scalability
* Long-term maintainability

across all credential services.

This governance applies to:

* Credential Registry
* Batch Management
* Certificate Generator
* Badge Generator
* Verification Publisher
* Wallet Export
* Metadata Builder
* Credential Intelligence Services
* Future Credential Operations modules

---

# Foundational Principle

Program Names may evolve.

Program Recognition may evolve.

Program Versions may evolve.

Program Codes must remain stable.

The ecosystem therefore treats Program Codes as the canonical institutional identifier.

---

# Program Code Definition

A Program Code is a unique institutional identifier assigned to a credential program.

Examples:

AOP

AIPA

AAIA

AAIP

AAIL

Program Codes exist to provide stability regardless of future program evolution.

---

# Program Code Authority Rule

Status: LOCKED

Program Codes are governed exclusively by Agile AI University.

Program Codes are institutional assets.

No operational system may independently create Program Codes.

No external user may define Program Codes.

---

# Program Code Uniqueness Rule

Status: LOCKED

Every Program Code must be globally unique.

Examples:

Valid:

AIPA

AAIA

AAIL

Invalid:

Duplicate use of an existing Program Code.

Program Codes must never collide.

---

# Program Code Immutability Rule

Status: LOCKED

Once assigned, a Program Code is immutable.

Examples:

AIPA

remains

AIPA

even if the program title changes.

Program Code changes are prohibited.

---

# Program Code Reuse Rule

Status: LOCKED

Retired Program Codes must never be reused.

Example:

AOP

Retired

↓

Unavailable Forever

Future programs must receive new Program Codes.

Historical integrity takes precedence over code recycling.

---

# Program Name Separation Rule

Status: LOCKED

Program Names and Program Codes are separate concepts.

Program Name

Human-readable

Program Code

System identifier

Examples:

Program Name:

Artificial Intelligence Professional Agilist

Program Code:

AIPA

Program Name modifications must not affect Program Codes.

# Program Lifecycle Rule

Status: LOCKED

Programs may transition through lifecycle states.

Examples:

* Active
* Retired
* Superseded
* Versioned

Lifecycle state does not alter Program Code identity.

Example:

AOP

Status:

Retired

Program Code:

AOP

remains unchanged.

---

# Program Successor Governance

Status: LOCKED

Programs may have successor relationships.

Examples:

AOP

↓

AIPA

AAIA-V1

↓

AAIA-V2

Successor mappings must be maintained through lifecycle metadata.

Successor relationships must never overwrite original Program Codes.

---

# Credential ID Governance

Status: LOCKED

Program Codes form part of credential identity.

Example:

AIPA-2026-000001

AAIA-2026-000145

The Program Code component must remain stable throughout the credential lifecycle.

---

# Batch Governance Integration

Status: LOCKED

Every batch must be associated with a Program Code.

Examples:

AIPA-2026-B001

AAIA-2026-B002

Batch governance relies upon Program Code stability.

Program Code association is irreversible once a batch is created.

---

# Template Governance Integration

Status: LOCKED

Certificate and badge rendering must use:

Program Code

↓

Template Key

↓

Template

Rendering systems must never depend on Program Names.

---

# Registry Authority Rule

Status: LOCKED

The Credential Registry is the authoritative repository for:

* Program Codes
* Program Names
* Credential Family Relationships
* Lifecycle Status
* Successor Relationships

No downstream system may redefine program information.

---

# Program Creation Governance

Status: LOCKED

Creation of a new Program Code requires institutional approval.

Minimum requirements:

* Program Name
* Program Code
* Credential Family
* Lifecycle State
* Template Mapping

Programs must be formally registered before operational use.

---

# Program Retirement Governance

Status: LOCKED

Retirement of a program does not remove the Program Code.

Retired programs remain permanently preserved within the registry.

Historical references must remain valid.

---

# Program Version Governance

Status: LOCKED

Program versioning must not require Program Code reuse.

Example:

Preferred:

AAIA

Version Metadata:

v1

v2

v3

rather than creating conflicting code structures.

Version management should occur through lifecycle metadata where possible.

---

# Operational Safety Rule

Status: LOCKED

Program Code selection must always be intentional.

Examples:

* No automatic selection
* No silent defaults
* No inferred program assignment

Program selection requires explicit administrator action.

This rule aligns with Batch Governance requirements.

---

# Future Evolution Rule

Status: LOCKED

The ecosystem must assume future expansion.

Examples:

* New credential families
* New professional credentials
* New executive credentials
* New micro credentials
* New recognition frameworks

Program Code governance must remain scalable regardless of ecosystem growth.

---

Program Codes are the canonical identifiers of the Agile AI University credential ecosystem.

Program Names communicate meaning.

Program Codes provide stability.

Credential lifecycles may evolve.

Recognition frameworks may evolve.

Program Codes preserve continuity across that evolution.

This document is governance-locked and applies across the entire Agile AI University credential ecosystem.
